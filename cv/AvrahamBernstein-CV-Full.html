    1. Most vehicles have 50-100 ECUs (i.e. small microprocessors) inside them. The ECUs are used to to control the engine, brakes, climate control, etc. These ECUs are connected to the main computer in the vehicle often via a very slow legacy CAN bus with an effective payload bandwidth of 128-512 kilo bits per second. It can literally take 2 or more hours to update all of the software on these ECUs even though the updates are zip compressed. Typically during the update process, the vehicle is immobilized.
    2. Historically for financial reasons, because the extra FLASH could add $25-50 to the cost of each vehicle, most manufacturers do not provide sufficient spare FLASH space on their ECUs to handle double banked A/B updates, where A is the original version and B is the new version. Therefore the update cannot take place in the background, and the vehicle is inoperable during the update process, because B will overwrite A. Often the update must take place at a vehicle service center, so someone, whether or not the owner or the manufacturer, must pay for the time that the vehicle sits in the service bay - easily $50-75 per hour.
    3. The delta compression algorithm is fundamentally different than the more well known zip compression algorithm. Zip compression searches for common strings, or byte sequences, in a file. It builds a dictionary of the common strings at the beginning of the encrypted file. The body of the encrypted file substitutes abbreviations for the dictionary entries, wherever the entry is encountered in the original file. Simplistically, assuming that the dictionary entries are at least 4 bytes long, but can abbreviated with just 2 bytes, then we can understand the net gain from using zip compression. Note that the zip algorithm requires only sufficient space in FLASH to store B, i.e. the updated file.
    4. However in the case of delta encoding, the algorithm searches for common strings in 2 different versions of a similar file, say version 1.0 and version 1.1. It may be the case that only a few bytes differ. The patch file contains the location and the values of the differences, i.e. the deltas. Zip compression of a binary executable typically can compress the original file by a factor of 2.5-3.0 times. But in the case of closely related versions of a similar file, the deltas can be over 10 times smaller.
    5. But the industry standard delta compression algorithm, bsdiff, requires enough space in FLASH to hold both the original A file, and at least enough space for a temporary copy of B, i.e. the updated file. But in the typically RAM and FLASH restricted environments found in vehicles, there is not enough memory to store a copy of both A and B, therefore the standard bsdiff algorithm is fundamentally infeasible. My invention was to tweak the algorithm to get around this limitation with nearly the same level of compression efficiency. Due to potential patent filing issues, I cannot (yet) describe here how I accomplished this feat. Note that I was in correspondence with the inventor, Colin Percival, of the FOSS (Free Open Source Software) bsdiff algorithm.
    6. I also experimented with Google’s FOSS Courgette delta compression algorithm. Unlike bsdiff compression which is instruction set agnostic, Courgette is based upon (a) having access to assembly source code with full symbol information, and (b) the target device must have an assembler and linker. Courgette usually generates updates that are much more compact than pure bsdiff. Google claims up to 7 times smaller. Google successfully uses Courgette to push updates for their Chromium and Android projects where there is sufficient room on the target to host a (mini) assembler and linker.
      • Prima facie having an assembler and linker on a target ECU device would be impossible due to the severe RAM and FLASH size constraints. However I invented a way to implement a table driven Courgette-like algorithm on a small ECU without the need for an assembler or linker, as long as a symbol table was available.

{
  const uint64_t z = (uint64_t)(x ^ rnd) | (uint64_t)rnd << 32;
  const uint64_t y = z ^ key;
  return y;
}

GCC_ATTRIB(const) CC_INLINE
uint32_t ayb_unwrap32(uint64_t key, uint64_t y)
{

  // without any knowledge of rnd
  const uint64_t z = key ^ y;
  const uint32_t rnd = z >> 32;
  const uint32_t x = rnd ^ (uint32_t)z;
  return x;
}

GCC_ATTRIB(pure,nonnull)
uint32_t adler32_core(const uint8_t *restrict data, int32_t n)
{
    // 1. The *core concept* of Adler-32 (derived from Fletcher)
    // is that the checksum is the weighted sum of all the bytes
    // in the data stream, where the weight is the byte's position
    // in the stream, i.e.
    //   sum = 1*data[0] + 2*data[1] + 3*data[2] + ... + n*data[n-1]
    //
    // 2. Even though Alder-32 is *extremely fast* to compute,
    // and is much stronger than the older LRC algorithm (i.e.
    // simply xoring all the bytes in the stream) which among
    // other deficiencies was not byte order dependent, Adler-32
    // is extremely *cryptographically weak* because for small n,
    // the sum will be much less than 2^32, therefore it will not
    // "spread" itself over the full bit range of sum.
    //
    // 3. Adler-32 is easy to "game", i.e. to generate "collisions".
    // For example when i < j < k
    //      sum = ... + data[i-1]*i + ... + data[j-1]*j + ...
    // or   sum = ... + data[i-1]*i + ... + data[j-1]*j + ... + data[k-1]*k + ...
    // It is not difficult to modify the data in order to generate
    // the same sum, namely
    //      sum = ... + data_prime[i-1]*i + data_prime[j-1]*j + ...
    //      etc.

    uint32_t sum = 0;

    for (int32_t i = 0; i < n; ++i) {
        sum += (uint32_t)data[i] * (uint32_t)(i+1);
    }

    return sum;
}

GCC_ATTRIB(pure,nonnull)
uint32_t ayb_checksum32_interim(
    const uint8_t *restrict data, int32_t n, int32_t k
)
{
    // Based upon the assumption that the expected value of each
    // each byte in the data stream is 128, 'k' should be designed
    // so that expected return value should be 2^32 - 1
    // in order to provide the best possible "spread".

    uint32_t sum = 0;
    uint32_t lrc = 0;

    for (int32_t i = 0; i < n; ++i) {
        // lrc makes it much more difficult to game, but not impossible
        lrc ^= (uint32_t)data[i];
        sum += lrc * (i+1);
    }

    if ((k & 1) == 0) {
        ++k; // force k to be odd
    }

    return sum*k;
}

GCC_ATTRIB(const)
CC_INLINE int32_t ayb_checksum32_calc_mult(int32_t n)
{
    // recall geometric sum of 1 + 2 + 3 + ... + n = n*(n+1)/2
    // ==> 128*(k + 2*k + 3*k + ... + n*k) = 2^32 - 1
    // ==> 128*k*n*(n+1)/2 = 2^32 - 1
    // ==> k*n*(n+1) ~= 2^26
    // ==> k ~= (2^26)/(n * (n+1))
    // ==> max n < 2^13, otherwise k is zero

    assert(n > 0 && n < 8192); // 2^13 = 8192

    int32_t k = (1L << 26)/(n * (n+1));

    if ((k & 1) == 0) {
        ++k; // force odd
    }

    return k;
}

GCC_ATTRIB(pure,nonnull)
uint32_t ayb_checksum32(const uint8_t *restrict data, int32_t n)
{
    const uint32_t k = ayb_checksum32_calc_mult(n); // has asserts

    uint32_t sum = 0;
    uint32_t lrc = 0;

    for (int32_t i = 0; i < n; ++i) {
        lrc ^= (uint32_t)data[i];
        sum += lrc * (i+1);
    }

    return sum*k;
}

      4. My most significant crytographic algorithm invention is a UHW (Uniformly distributed Hamming Weight) implemented with DFTs (Digital Fourier Transformations). A standard HW (Hamming Weight) is the sum of the 1 bits in a binary number. Therefore the range of a HW value is from zero to the number of bits in the number, e.g. typically 32 or 64 depending upon the CPU architecture. The range of a UHW is the same. Standard HWs are non-linear functions, but they are not cryptographically useful/interesting on account of their underlying binomial distribution. Rotating or permuting the bits in a number according to its HW does not change the HW, therefore the operation is invertible. But a UHW is also non-linear. It is also invariant with respect to bit rotation and permutation, and therefore is invertible as long as the rotation/permutation uses groups the size of 1/4 of the bits in the underlying value, e.g. 32-bit and 64-bit values. Cryptographic operations based upon UHWs are much stronger than the DDRs (Data Dependent Rotations) permutations made famous by Rivest in his RC5 cipher. For example in RC5 the low 5 bits of the preceding 32-bit value is used to rotate the subsequent 32-bit value. But by using UHWs, the UHW of the preceding 64-bit value could be used to rotate the subsequent 64-bit value. The UHW has many properties similar to standard HWs. I will publish a paper on the algorithm in the near future, including an empirical proof that the distribution is uniform (but with some minor perturbations). Here is the source code of the core algorithm:
      #include <stdint.h>

#ifdef __cplusplus
#define CC_INLINE       inline
#else
#define CC_INLINE       static inline
#endif

#ifdef __GNUC__
#define GCC_ATTRIB(...) __attribute__((__VA_ARGS__))
#else
#define GCC_ATTRIB(...)
#endif

GCC_ATTRIB(const)
int ayb_uhw_depth_32(uint32_t x, const int depth)
{
    #undef BIT_WIDTH
    #define BIT_WIDTH (sizeof(x)*CHAR_BIT)

    assert(depth > 0 && depth <= BIT_WIDTH/2 && IS_POW2(depth));

    // The fundamental idea of the algorithm is to successively
    // divide x into 1-bit, 2-bit, 4-bit, ..., and depth-bit
    // groups where depth is a power of 2, and the max value of
    // depth <= half of the bits in x.

    // When the value of a bit group is greater than half of its
    // maximum value then the group sum, i.e. sum[i] is
    // incremented by 1.
    // The total sum = sum[0]*1 + sum[1]*2 + sum[2]*4 ...

    // The "secret sauce" is: uhw = sum % (BIT_WIDTH + 1)

    // Clearly rotating the bits in x (or permuting it) in groups
    // of depth bits does not change the UHW.

    int sums[LOG2(BIT_WIDTH)] = { popcount(x) };

    int i = 0;
    int shift = 2;
    typeof(x) mask = (1 << shift) - 1;

    for (; shift <= depth; shift <<= 1, mask = (1 << shift) - 1, ++i) {
        for (typeof(x) y = x; y; y >>= shift) {
            sums[i+1] += (y & mask) > (mask/2);
        }
    }

    int sum = sums[0];

    for (; i > 0; --i) {
        sum += sums[i] << i;
    }

    const int uhw = sum % (BIT_WIDTH + 1);
    // uhw is in the range [0,sizeof(x)*BIT_WIDTH]
    return uhw;
}

GCC_ATTRIB(const) CC_INLINE
int ayb_uhw32(uint32_t x)
{
    // most efficient depth (in bits) is sizeof(x)*2,
    // i.e. 1/4 of the bit width of x

    return ayb_uhw_depth_32(x, sizeof(x)*2);
}

    3. The above algorithms relied upon (1) the industry standard SVG (Scalable Vector Graphics) package found on all modern web browsers, and (2) sound cards found on all PCs and smart phones. Ideally the blind user should navigate with a graphics tablet and stylus (typically available for under $100), or a device with a touch screen. Blind people have accurate hand to mind’s-eye coordination, in the sense that they can easily navigate a graphics tablet, i.e. to the corners, the edges, and to the middle. However a mouse is very difficult for them to use because when moving it beyond the edge of the display, the cursor stops even though the mouse keeps moving. They lack the visual feedback required to detect the mouse cursor position, which is not the case when a graphics tablet is programatically defined to be a rectilinear mapping into the display.
      1. The core sound inventive technique was that there was always background audio feedback at the cursor position. The display was conceptually divided into a grid of 5 (or more depending upon the audio acuity of the user) 12-note (chromatic) octaves in the vertical and horizontal direction. The vertical axis was assigned one instrument, say a piano, while the horizontal axis was assigned a second instrument, say a flute. Therefore the user would always hear a low volume combination of notes at the current cursor position.
      2. Interesting objects on the display would use the same audio cursor technique, but at a higher level of volume. Say one is tracing a circle whose circumference (a few pixels wide) is indicated in white, while the rest of the screen has a black background. When the cursor is over the circumference the volume would be much louder than when over the background.
      3. Similarly when searching for the point where two lines intersect, as the cursor approached the intersection point, the volume would increase.
      4. By defining the screen objects with SVG, there can be a seamless zoom-in and zoom-out. By default the graphics tablet would be rectilinear mapped to the current window dimensions.
      5. In the case of geographic maps, say a topographical map, to display elevation, the user could redefine the isotherm granularity, e.g. every 10 meters or every 100 meters. And the user could help to define his own color legend, which could be mapped to different instrument combinations.
    4. I invented an algorithm for automatically modifying fonts that makes them much easier for dyslexics to recognize. It is not effective to simply place a dot at the bottom of each letter like sixes and nines are often differentiated for normal readers because (1) dyslexics typically can flip/reverse letters across the middle vertical axis, and (2) without extending the upper and lower margins bordering each letter, when viewing lines of text where every lettered is dotted, it would be extremely difficult to determine whether or not a dot is above or below a letter. Instead my invention is based upon a similar principle used for machine vision identification of a QR code. Namely a QR code has 3 relatively large black squares in 3 of the 4 corners, i.e. in the top-left, top-right, and bottom-left, but not the bottom-right. Therefore it is trivial for an image processing application to rotate the QR code into its canonical position. So now if we automatically widen the left and right margins of a character with an identical amount of padding on each side, and conceptually divide the left margin in half with a vertical line, and place a small dot in the middle of the inner half of the expanded left margin, then the single simple rule for a dyslexic reader, is that there must be a dot in middle of the the inner half of the left margin.
